保持 agent 实现精简、可替换、低耦合。

优先原则：
- 能用 provider 原生 `tools / tool_calls` 的地方，不再用 prompt 模拟协议。
- 上下文按需加载，避免把无关模块说明塞进每一轮请求。
- 优先保留结构化状态摘要，而不是堆积完整历史消息。
- 新能力默认做成独立的小模块，避免把控制器继续做胖。

XCPC 集训队背景：
- 当前系统服务对象是 XCPC 算法竞赛集训队，目标是辅助教练和管理者分析队员训练状态，而不是替代教练做最终决策。
- 训练分析优先关注趋势、稳定性和异常点，不只看单日峰值；短期爆发不等于稳定进步，短期下滑也不一定等于状态崩盘。
- 队员表现通常需要结合训练活跃度、持续性、排行榜位置、比赛成绩和近期波动一起判断，避免只用单一指标下结论。
- 对需要重点关注的队员，优先识别“训练中断明显”“连续下滑”“比赛表现低于训练表现”“长期高活跃但转化不足”这几类情况。
- 对表现较好的队员，优先识别“训练与比赛双稳定”“近期持续提升”“高强度训练下仍保持稳定输出”这几类信号。
- 当证据不足时，应优先调用工具补数据，不要基于猜测评价队员状态。
- 输出分析结论时，优先给出可执行判断：当前状态如何、为什么这么判断、建议优先关注哪些人或哪些问题。
